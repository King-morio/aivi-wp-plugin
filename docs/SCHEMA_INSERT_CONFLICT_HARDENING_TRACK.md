# Schema Insert Conflict Hardening Track

Last updated: 2026-03-24

## Purpose

This track hardens AiVI's schema insert flow so JSON-LD assists stop behaving like blind
append operations.

The goal is to make schema insert behavior safe and predictable across:

- editor-inserted AiVI schema blocks
- existing manual JSON-LD already in the post
- schema generated by SEO plugins or themes outside the editor block tree
- repeat inserts across reruns and later editing sessions

## Current Diagnosis

- current insert is append-only:
  - AiVI creates a new `core/html` block with a JSON-LD `<script>` and inserts it at the end
    of the editor content
- duplicate protection is only session-local:
  - the current fingerprint guard only blocks the same draft from being inserted again during
    the current run/session
- most insertable schema assists are fresh seeds:
  - `article_jsonld`
  - `faq_jsonld`
  - `howto_jsonld`
  - `itemlist_jsonld`
- only `schema_alignment_jsonld` currently reuses an existing schema object as a base
- preflight already knows about rendered-page JSON-LD, but the editor insert path does not
  consult that information before appending

## Guardrails

- Do not implement blind merge behavior.
- Prefer `detect -> classify -> no-op / replace / copy-only / append`.
- Automatic replace is allowed only when AiVI can identify the target confidently.
- Existing schema outside editor blocks should never be silently overwritten.
- If any milestone needs a file outside its declared write set, stop and amend this track first.

## Adopted Reconciliation Decisions

- `jsonld_safe_insert` needs real conflict awareness, not just append-at-end safety.
- AiVI should inspect both:
  - current editor JSON-LD blocks
  - rendered-page / manifest JSON-LD already detected during analysis
- the insert path should choose one of four outcomes:
  - `no_op_existing_match`
  - `replace_existing_ai_block`
  - `copy_only_external_conflict`
  - `append_new_block`
- `schema_matches_content` is the safest first candidate for replace/update behavior because it
  already derives from existing schema when available.
- `article_jsonld`, `faq_jsonld`, `howto_jsonld`, and `itemlist_jsonld` should remain
  conservative until target confidence is explicit.

## Milestones

### M1 - Existing Schema Discovery Layer

**Status:** Complete

**Goal**

Teach the editor-side insert flow to discover existing JSON-LD blocks and classify whether
they look AiVI-managed, manual, or external to the editor.

**Planned write set**

- `assets/js/aivi-overlay-editor.js`
- `tests/js/overlay-schema-assist.test.js`

**Scope**

- scan current Gutenberg blocks for `core/html` JSON-LD script blocks
- parse valid JSON-LD where possible
- extract lightweight comparison signatures:
  - schema type
  - name/headline
  - url / mainEntityOfPage
  - FAQ question names
  - HowTo step names
  - ItemList item names
- add AiVI-managed block markers for future inserts

**Out of scope**

- builder changes
- serializer changes
- automatic replace behavior

**Outcome**

- the overlay editor now scans current Gutenberg blocks for `core/html` JSON-LD script blocks
  before insert attempts
- valid JSON-LD blocks are now parsed into lightweight comparison signatures covering:
  - schema type
  - name/headline
  - url / mainEntityOfPage
  - FAQ question names

  - ItemList item names

  - `aivi_managed`
  - `manual`
- newly inserted AiVI schema blocks now carry an explicit AiVI marker comment so later
  milestones can distinguish managed inserts from manual editor HTML
- current insert policy is intentionally unchanged in `M1`:
  - no-op / replace / copy-only decisions are still deferred to `M2`
  - same-session duplicate blocking remains the only active insert guardrail

### M2 - Conflict Classification And Insert Policy

**Status:** Complete

**Goal**

Replace blind append behavior with a conflict-aware decision engine before any insert happens.

**Planned write set**

- `docs/SCHEMA_INSERT_CONFLICT_HARDENING_TRACK.md`
- `assets/js/aivi-overlay-editor.js`
- `tests/js/overlay-schema-assist.test.js`

**Scope**

- add policy outcomes:
  - `no_op_existing_match`
  - `replace_existing_ai_block`
  - `copy_only_external_conflict`
  - `append_new_block`
- keep replace behavior limited to high-confidence matches only
- keep low-confidence collisions out of auto-insert
- preserve current same-session fingerprint protection as a second guardrail

**Out of scope**

- backend draft generation changes
- manifest/runtime contract changes

**Outcome**

- the overlay insert path now resolves a conflict-aware policy before any JSON-LD block is
  inserted into the editor
- the policy now chooses one of four outcomes:
  - `no_op_existing_match`
  - `replace_existing_ai_block`
  - `copy_only_external_conflict`
  - `append_new_block`
- policy decisions now consider both:
  - existing editor JSON-LD discovered from current Gutenberg blocks
  - rendered-page / manifest JSON-LD already captured during analysis
- identical existing schema now short-circuits to a no-op instead of creating another block
- a single clear AiVI-managed target can now be replaced in place instead of duplicated
- low-confidence or external/manual conflicts now switch the assist to copy-only behavior
  instead of silently appending a competing schema block
- same-session fingerprint protection remains active as a second guardrail after policy
  resolution
- this milestone stayed editor-local:
  - no backend builder changes
  - no serializer changes
  - no runtime-contract changes

## Acceptance Targets

- clicking `Insert schema` does not blindly append duplicate JSON-LD when equivalent schema is
  already present
- equivalent existing AiVI-managed schema can be replaced cleanly instead of duplicated
- third-party or theme/plugin-provided schema outside the editor does not get silently collided
  with or overwritten
- schema assist status clearly tells the user whether AiVI inserted, replaced, skipped, or
  switched to copy-only mode
- later reruns and reloads do not lose all duplicate awareness
